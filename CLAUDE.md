# OwnBoard — Repo Guide for AI Agents

Hackathon project (OpenAI Codex x NamasteDev, 4 days). Turns onboarding into something verifiable: policy quizzes, git-history-grounded codebase quizzes, a real (non-self-reported) skill graph with bus-factor detection, and LLM-drafted expert routing. Everything is citation-grounded; low-confidence answers escalate to a human instead of hallucinating.

Full product spec: [`docs/2026-07-16-onboard-platform-prd.md`](docs/2026-07-16-onboard-platform-prd.md). Router/service docstrings cross-reference PRD sections (e.g. `§6.3`) — read the cited section before touching that domain's logic.

## Layout

| App | Path | Stack | Local URL |
|-----|------|-------|-----------|
| Backend API | [`backend/`](backend/CLAUDE.md) | FastAPI + SQLAlchemy async + Postgres/pgvector | http://localhost:8000 (docs at `/docs`) |
| Frontend | [`frontend/`](frontend/CLAUDE.md) | Next.js 15 App Router + React 19 | http://localhost:3000 |

Read the sub-directory `CLAUDE.md` before working in that app — this file only covers what's shared. Frontend also has [`frontend/DESIGN.md`](frontend/DESIGN.md) for visual/UX consistency — read it before touching any UI.

The browser never calls FastAPI directly: `frontend` proxies every request through its own `app/api/**/route.ts` handlers to the backend. Keep that boundary intact when adding a new domain — a new backend endpoint needs a matching Next.js route handler before the UI can call it.

## Running everything

```bash
make setup   # first time: env files, deps, Postgres, migrations
make dev     # backend + frontend together
```

`make help` lists all targets, including running each app independently (`make backend`, `make frontend`) and DB controls (`make db-up`, `make db-down`, `make db-reset`, `make migrate`). See [`README.md`](README.md) for env file details.

## Current implementation state

Only **repo** and **employee** CRUD are wired end-to-end. Every other domain (`skill_graph`, `rag`, `quiz`, `chat`/archaeology, `expert_routing`, `dashboard`) already has its router, schema, and service skeleton in place, but service methods raise `NotImplementedError` and the API answers `501` until implemented — this is intentional scaffolding, not a bug. The frontend already has pages/hooks wired against these stub contracts, so filling in a backend domain should make the corresponding UI work without frontend changes, unless the response shape changed.

## Conventions that span both apps

- IDs are opaque strings everywhere (20-char random, generated by the backend) — never assume integer/auto-increment IDs in frontend code. The one exception is `Organization.id`, which is the Clerk organization id verbatim (not backend-generated), since every other tenant-owned table's `org_id` FK needs to match it exactly.
- Zod schemas in `frontend/src/schemas/*.schema.ts` are the frontend's source of truth for API response shapes, independent from the backend's Pydantic schemas — if you change a backend response shape, update the matching zod schema (and its `snake_case`→`camelCase` `.transform()` if present).
- Auth and multi-tenancy are real, via Clerk Organizations — see [`backend/CLAUDE.md`](backend/CLAUDE.md)'s Conventions section. Every company is a Clerk Organization; every domain (policies, quizzes, assignments) is scoped to `org_id`, which is always derived server-side from the caller's verified session/active-org, never from client input. The frontend enforces an active org via `src/middleware.ts` (redirects to `/select-organization` when missing) and exposes Clerk's `OrganizationSwitcher` in the header — don't build a custom org-switching UI.
